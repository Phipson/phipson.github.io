Voice Interactions for Reality Composer

Overview

Timeline

4 Weeks (1 Month)
Auugst 2019 - September 2019

Involved members

Phipson (Me)/ Lead
Other 3D Software Engineers (undisclosed)

Project Content

Voice library; ARKit in iPad; User Study; Programming with Voice


Apple's newly released Reality Composer offers new features for end-users to animate and interact with virtual content using ARKit. However, how would Reality Composer look like in an AR environment? Towards the end of my summer internship at Apple, I worked with a group of 3D software engineers at Apple's Applied Research Team to explore and investigate ways we could use voice to add actions and logic into Reality Composer.

In contrast to my original logic programming interface, which used 3D hand gestures as a primary mode of input, our goal was to identify ways to program strictly using voice, and design a scalable representation of behaviors and actions in Reality Composer.

Currently, Reality Composer does not offer ample utilities to efficiently prototype and debug projects in a 3D environment. As a result, users have to navigate the logic of their program using typical 2D gestures or keyboard and mouse interface. On the other hand, voice offers users more expressive capabilities that are familiar and intuitive for casual end users with little programming experience. Rather than representing an action or behavior in terms of a programming language, we propose using voice input to generate code by saying the intended action to be implemented.


(As my work is under Apple's NDA, I am not allowed to describe the project in detail. However, my work was demoed live at internal Open House events, and videos of my work have been showcased across different teams at Apple)

Design Process

User Profiles and Use Cases

To define the scope of our project, we first brainstormed different possible use cases for Reality Composer. We then categorized different these scenarios based on target audience and the intended product. For preliminary research purposes, we then selected a small portion of the categories most representative of our target audience as the basis of our research. The following user profiles resulted from the brainstorming session:

Bowling Ball Game
Virtual Apple Store
AR Fast Food Menu

To represent the logic for each of the scenarios, we went with a bottom-up approach: we distilled the common components of each logical command in Reality Composer, and from there, identified the types of possible voice commands that users could utter to express these components. After our brainstorming sessions, we derived 3 classes of data we could use to represent behaviors in Reality Composer.

In tandem with our brainstorming sessions, we also designed and are currently piloting a user study investigating voice commands users would use to represent a program in Reality Composer.

Designing an Interface

To give users visual feedback of their voice input, we proceeded by investigating different representations of the logic. One constraint we have is that the visual representation of the logic must convey the underlying modes of interaction: as our research is focused on voice input, we must avoid visuals that suggest other inputs like hand gestures are the sole method used to interact with the system.

Taking this into consideration, my goal was to eschew block or graph-based representations for our interface, which led me to a card-based representation of our logic, as inspired by the following concepts.

Google Game Builder: Offers a 2D UI for card-based representation of logic. Having both a graphic and textual representation of the logic is very digestible and easy to understand. Also steers away from the complex node-graph, block-based representation. Only caveat is that it is a very 2D styled visualization method, and is thus not suited for 3D visualizations.

Playing Cards: Observing physical use of cards, I noticed that playing cards provide an improvement to Google's Game Builder card representation, as it uses the design of the card itself to thematically categorize cards. Numeric values on the card can also be used to reflect certain elements of the card-specific game, which we can use in context to Reality Composer for representing action-related numbers and values.

Following several iterations, I created 3 virtual cards, each categorized by their color, to represent the 3 aforementioned categories. Each card contained both a text description corresponding to the voice input the users would use to generate the card, as well as a visual graphic representing the type of information stored in the card. This allows users to associate the form of the card with its function, thereby making the programming process more efficient.

Unlike the logic block representation of the program, I added features to expand and collapse details in each card. By using hand gestures as a secondary mode of input, users could hide and show certain elements of the card. Note that hand gestures cannot be used to generate logic, so voice is still considered a primary mode of input.

Organizing the Workspace

One key feedback I received from my original visual programming prototype was that I should consider ways to organize and separate the interaction environment from the programming environment: because users are tasked with programming and interacting in the same 3D environment, as the programs grow increasingly complex, without organization utilities, users will easily confused themselves as they add more logic.

Taking this into consideration, I designed a central system to collapse all the logic cards. The idea is that by using voice commands, users can hide and reveal certain elements in the logic hierarchy, thus setting the scope of their workspace. Although I was not able to implement the system, I did add code and mockups showing how it could be done.

Final Thoughts


Unfortunately, I was not able to continue working on the project, as I was nearing the end of my internship. However, in contrast to my original visual programming project, which was an independent research exploration, I learned more about how to coordinate and collaborate with other researchers in teams, and ultimately gained a lot of experience providing and receiving constructive feedback from my colleagues.
